\chapter{Marco conceptual}

\section{Revisión de la literatura}

El desarrollo de un sistema administrativo para farmacia requiere fundamentos de arquitectura de software, aplicaciones web, persistencia transaccional y seguridad de acceso. En sistemas empresariales, la arquitectura no se limita a seleccionar tecnologías, sino a definir responsabilidades, dependencias y mecanismos de evolución. \textcite{richards2015architecture} describe la arquitectura en capas como un patrón apropiado cuando se busca separar presentación, lógica de negocio y acceso a datos. Esta separación resulta pertinente para el caso de estudio porque las operaciones farmacéuticas combinan interacción de usuario, validaciones de dominio y persistencia histórica.

La gestión de datos transaccionales también es central. \textcite{kleppmann2017data} enfatiza que los sistemas de datos deben diseñarse considerando confiabilidad, mantenibilidad y consistencia. En una farmacia, estos atributos se relacionan directamente con compras, lotes, vencimientos y movimientos de inventario. Una compra recibida no debe entenderse solamente como un registro administrativo, sino como el origen de existencias que podrían ser vendidas, auditadas o anuladas bajo reglas controladas.

Para la interfaz web, \textcite{banks2020learning} presenta React como una herramienta orientada a construir interfaces por componentes y administrar cambios de estado de forma declarativa. Esta orientación permite construir pantallas administrativas donde filtros, formularios, estados de carga y confirmaciones de operación se mantienen comprensibles para el usuario. En el lado servidor, \textcite{mardan2014express} expone Express como un marco flexible para construir APIs HTTP con rutas y middleware, lo que facilita separar autenticación, autorización y tratamiento de errores. De forma complementaria, \textcite{casciaro2020node} desarrolla patrones de diseño para aplicaciones Node.js que respaldan la modularidad y la separación de responsabilidades en servicios web.

La persistencia relacional se sustenta en PostgreSQL, cuyo modelo permite restricciones, transacciones, índices y relaciones entre entidades \parencite{postgresql2026docs}. Para el acceso a datos desde TypeScript, Prisma ofrece un ORM con esquema declarativo, migraciones y cliente tipado \parencite{prisma2026docs}. Esta combinación es adecuada cuando se necesita representar usuarios, roles, proveedores, compras, productos, unidades, lotes y movimientos sin perder integridad referencial.

La validación de datos y los contratos compartidos se apoyan en TypeScript y Zod. TypeScript aporta tipado estático para reducir errores de integración entre partes del sistema \parencite{typescript2026handbook}, mientras que Zod permite expresar esquemas de validación ejecutables \parencite{zod2026docs}. En sistemas administrativos, esta práctica reduce divergencias entre lo que la interfaz envía, lo que la API acepta y lo que la base de datos puede conservar.

Finalmente, la seguridad de sesión se relaciona con autenticación, autorización y auditoría. El estándar JSON Web Token define un formato compacto para transmitir declaraciones verificables entre partes \parencite{rfc7519}. Sin embargo, su uso debe complementarse con controles de rol, expiración, validación de usuario activo y registro de acciones. En el contexto farmacéutico, la auditoría no es un adorno técnico: permite reconstruir quién creó, recibió o anuló una operación que afecta inventario.

\section{Conceptos, teorías y modelos}

\subsection{Sistema de información transaccional para farmacia}

Un sistema de información transaccional registra operaciones que modifican el estado de una organización. En una farmacia, estas operaciones incluyen compras, ventas, movimientos de inventario, administración de productos, usuarios y auditoría. La particularidad farmacéutica se encuentra en la trazabilidad: el sistema debe conservar datos suficientes para explicar el origen, disponibilidad, costo y estado de cada existencia relevante.

El concepto de lote se refiere a un conjunto identificable de productos que comparten una referencia de fabricación o distribución. La fecha de vencimiento indica el límite temporal de uso o comercialización segura. La unidad comercial puede diferir de la unidad base de inventario; por ejemplo, una compra puede registrarse por caja, pero el control interno puede requerir unidades individuales. Por esta razón, el sistema debe manejar factores de conversión, cantidades base y costos base.

\subsection{Arquitectura web modular}

La arquitectura web modular separa el sistema en responsabilidades comprensibles: una interfaz orientada al usuario, una API que aplica reglas de negocio, contratos de datos que definen entradas y salidas, y una base relacional que conserva el estado histórico. Esta separación permite que una pantalla cambie sin reescribir reglas de recepción de compras, y que una regla de inventario evolucione sin duplicarse en cada vista.

\subsection{API REST y control de roles}

Una API REST organiza recursos y operaciones mediante HTTP. Para el caso de estudio, proveedores y compras se tratan como recursos administrativos; la recepción y la anulación se modelan como transiciones de estado controladas. La autorización por roles permite distinguir usuarios con capacidad administrativa de usuarios de venta. Esta distinción es importante porque una compra recibida puede crear inventario y afectar costos, por lo que no debe estar disponible para todo perfil de usuario.

\subsection{Persistencia transaccional e inventario por capas}

La persistencia transaccional asegura que una operación crítica se complete de manera íntegra o no se aplique. En una compra recibida, esto significa que el cambio de estado de la compra, la creación de capas de inventario, los movimientos asociados y la auditoría deben mantenerse coherentes. El inventario por capas permite conservar existencias separadas por origen de compra, costo base, lote y vencimiento, incluso cuando dos compras correspondan al mismo producto.

Este enfoque favorece la trazabilidad farmacéutica porque permite explicar no solo cuánto stock existe, sino de dónde proviene y bajo qué costo ingresó. También facilita una futura visualización de kardex o stock por lote sin sacrificar la precisión histórica de las operaciones.

\subsection{Validación compartida y mantenibilidad}

La validación compartida consiste en expresar reglas de datos en contratos reutilizables por más de una parte del sistema. En vez de depender únicamente de validaciones visuales, el sistema define restricciones para campos como NIT, estado de proveedor, fecha comercial, estado de compra, cantidad, costo, lote, vencimiento y motivo de anulación. Esta práctica mejora la mantenibilidad porque reduce interpretaciones contradictorias de los mismos datos.

\section{Antecedentes históricos o contextuales}

Los sistemas de punto de venta evolucionaron desde registros centrados en importes hacia plataformas que conectan ventas, inventario, proveedores, usuarios y reportes. En farmacias, esta evolución es especialmente relevante porque la mercadería puede caducar, requerir control por lote o depender de proveedores con condiciones comerciales distintas. Por tanto, el problema no consiste únicamente en vender productos, sino en mantener consistencia entre transacciones comerciales, existencias físicas y evidencia administrativa.

En el contexto tecnológico actual, las aplicaciones administrativas tienden a separar cliente y servidor mediante APIs, utilizar bases de datos relacionales para información crítica y aplicar mecanismos de autenticación con roles. Esta orientación permite construir sistemas incrementales sin perder control sobre la consistencia de datos. Para una farmacia, el flujo de proveedores y compras recibidas constituye una base estratégica: antes de vender con confianza, el sistema debe saber cómo ingresó la mercadería, con qué costo, bajo qué lote y qué usuario autorizó la recepción.

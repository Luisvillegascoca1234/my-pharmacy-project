\chapter{Marco conceptual}

\section{Revisión de la literatura}

El desarrollo de un sistema de punto de venta para farmacia requiere articular fundamentos de ingeniería de software, arquitectura de aplicaciones web, seguridad de autenticación y gestión persistente de datos. Desde la ingeniería de software, el sistema debe entenderse como un producto compuesto por requisitos, diseño, implementación, pruebas y mantenimiento, no solamente como una colección de pantallas o funciones aisladas. \textcite{pressman2020software} sostienen que la calidad del software depende de procesos técnicos sistemáticos, mientras que \textcite{sommerville2016software} enfatiza la importancia de estructurar el sistema para controlar complejidad, evolución y mantenibilidad.

En aplicaciones empresariales, la separación de responsabilidades es un principio recurrente. \textcite{fowler2003patterns} describe patrones de arquitectura de aplicaciones empresariales orientados a distinguir presentación, lógica de dominio y acceso a datos. Esta división es pertinente para el presente proyecto porque el código fuente evidencia una organización en capas: una interfaz web desarrollada con React y Vite, una API HTTP construida con Express y TypeScript, y una capa de persistencia relacional basada en PostgreSQL accedida mediante Prisma ORM.

La literatura y documentación técnica también respaldan el uso de interfaces web de una sola página, contratos compartidos y validación de datos. React se orienta a la construcción de interfaces declarativas por componentes \parencite{react2026learn}, Vite proporciona un entorno de desarrollo y construcción para aplicaciones frontend modernas \parencite{vite2026guide}, y TypeScript añade tipado estático sobre JavaScript para reducir errores de integración en tiempo de desarrollo \parencite{typescript2026handbook}. En este contexto, Zod se emplea como mecanismo de validación de esquemas y de inferencia de tipos, lo que permite alinear los datos enviados por el cliente con los datos aceptados por el servidor \parencite{zod2026docs}.

Respecto a la seguridad de sesión, el proyecto implementa autenticación basada en credenciales, hashing de contraseñas, emisión de tokens JWT y autorización mediante el encabezado HTTP \texttt{Authorization: Bearer}. El estándar JSON Web Token define un formato compacto para representar declaraciones entre partes de forma verificable \parencite{rfc7519}. Aunque JWT no reemplaza por sí solo una política integral de seguridad, su uso en una API HTTP moderna permite desacoplar la sesión del almacenamiento de estado del servidor, siempre que se protejan adecuadamente la clave de firma, el tiempo de expiración, el transporte HTTPS y la revocación o renovación cuando el dominio lo exija.

Finalmente, el uso de Docker en el proyecto responde a una necesidad de reproducibilidad del entorno. Docker permite empaquetar y ejecutar servicios en contenedores aislados, reduciendo diferencias entre equipos de desarrollo y facilitando la provisión de dependencias como PostgreSQL \parencite{docker2026overview}. En este caso, \texttt{docker-compose.yml} define el servicio de base de datos PostgreSQL 16 con volumen persistente y verificación de salud, por lo que el contenedor no constituye la aplicación completa, sino la infraestructura local que sostiene la capa de datos.

\section{Conceptos, teorías y modelos}

\subsection{Sistema de información transaccional para farmacia}

Un sistema de información transaccional registra operaciones cotidianas que modifican el estado de una organización: ventas, compras, pagos, movimientos de inventario y auditoría. En el dominio farmacéutico, esta clase de sistema exige trazabilidad mayor que un catálogo simple, porque el inventario depende de lotes, vencimientos, proveedores, precios, usuarios autorizados y registros históricos. Por ello, el diseño conceptual debe considerar que cada operación crítica no solo produce una respuesta visual al usuario, sino también cambios persistentes y verificables en la base de datos.

El proyecto se orienta a un punto de venta farmacéutico. Sus lineamientos técnicos contemplan módulos como autenticación, usuarios, roles, productos, unidades, proveedores, compras, inventario, ventas, caja, facturación, devoluciones, auditoría, alertas, reportes y exportaciones. Esta división funcional permite modelar el sistema como un monolito modular: una sola aplicación backend desplegable, pero internamente organizada por dominios para evitar acoplamiento excesivo.

\subsection{Arquitectura de tres capas}

La arquitectura de tres capas separa el sistema en presentación, lógica de aplicación o negocio, y persistencia. Esta separación no debe interpretarse solamente como carpetas distintas, sino como una distribución de responsabilidades. La capa de presentación interactúa con el usuario y muestra estados; la capa de negocio valida reglas, coordina operaciones y aplica políticas; la capa de datos encapsula consultas, transacciones y persistencia. Esta organización mejora la mantenibilidad porque una modificación visual no debería exigir cambios directos en consultas de base de datos, y una modificación del motor de persistencia no debería alterar los componentes de interfaz \parencite{sommerville2016software,fowler2003patterns}.

\subsection{Frontend web moderno}

El frontend del sistema utiliza React como biblioteca de interfaz y Vite como herramienta de desarrollo y construcción. Conceptualmente, esta combinación corresponde a una aplicación web de una sola página, donde el navegador carga una aplicación cliente que actualiza vistas sin solicitar documentos HTML completos en cada interacción. React facilita la composición de componentes reutilizables y Vite acelera el ciclo de desarrollo mediante servidor local y empaquetado optimizado \parencite{react2026learn,vite2026guide}.

El estado de autenticación se administra con Zustand. En el código revisado, el almacén del módulo de autenticación conserva el token y restaura la sesión al iniciar la aplicación. Este patrón evita pasar manualmente credenciales por múltiples componentes, pero exige disciplina: el estado global debe reservarse para información transversal y no para datos efímeros de pantalla. El cliente HTTP base centraliza la URL de la API, serializa el cuerpo de las solicitudes y agrega el encabezado \texttt{Authorization} cuando existe token.

\subsection{Backend, API REST y monolito modular}

El backend está construido con Express y TypeScript. Express proporciona el enrutamiento HTTP y el manejo de middleware, mientras que TypeScript mejora la legibilidad y seguridad del código mediante tipos explícitos \parencite{express2026guide,typescript2026handbook}. La organización observada responde a un monolito modular: los módulos se agrupan por dominio y cada uno puede contener rutas, controladores, servicios, repositorios y tipos.

El patrón controller-service-repository delimita responsabilidades. El controlador recibe la solicitud HTTP, valida la entrada y delega; el servicio concentra reglas de aplicación, autenticación, auditoría y coordinación; el repositorio encapsula el acceso a Prisma y, por tanto, a la base de datos. Esta división reduce el acoplamiento entre HTTP, reglas de negocio y persistencia, lo que es coherente con los criterios de modularidad y mantenibilidad descritos por la literatura de ingeniería de software \parencite{pressman2020software,sommerville2016software}.

\subsection{Persistencia relacional y ORM}

PostgreSQL es el sistema de gestión de base de datos relacional definido para el proyecto. Una base relacional resulta adecuada cuando se requiere integridad entre entidades como usuarios, roles, permisos, auditoría, productos, ventas y movimientos. PostgreSQL documenta características de transacciones, tipos de datos e integridad que son relevantes para aplicaciones empresariales \parencite{postgresql2026docs}.

Prisma ORM se utiliza como capa de acceso a datos. Su esquema declara modelos, relaciones, restricciones y migraciones; el cliente generado permite consultar la base desde TypeScript \parencite{prisma2026docs}. En términos conceptuales, Prisma no elimina la necesidad de arquitectura: el ORM debe permanecer en repositorios o infraestructura, de modo que la lógica de negocio no quede mezclada con detalles de consulta.

\subsection{Autenticación moderna basada en tokens}

El flujo de autenticación implementado responde a una estrategia moderna de APIs HTTP. El usuario envía correo y contraseña; el backend normaliza el correo, busca al usuario, compara la contraseña con un hash mediante \texttt{bcryptjs}, verifica el estado de la cuenta, registra auditoría y emite un JWT firmado. En solicitudes posteriores, el cliente envía el token en el encabezado \texttt{Authorization: Bearer}. El middleware del backend verifica el token, obtiene al usuario activo y adjunta la identidad autenticada a la solicitud.

Este flujo separa autenticación y autorización. La autenticación responde a la pregunta de quién es el usuario; la autorización determina qué puede hacer, a partir de roles y permisos. El modelo de datos incluye usuarios, roles, permisos y una relación de asignación de permisos a roles. Además, el registro de auditoría para inicios de sesión exitosos, fallidos y cierres de sesión añade trazabilidad, aspecto importante en sistemas administrativos donde las acciones deben poder reconstruirse.

Desde una perspectiva académica, JWT debe tratarse como un mecanismo de representación de sesión, no como una garantía absoluta de seguridad. La protección depende de prácticas complementarias: secreto robusto, expiración limitada, transporte seguro, validación de usuario activo y auditoría. En el proyecto, la expiración del token se parametriza mediante variables de entorno y la validación del usuario activo evita aceptar tokens de cuentas deshabilitadas.

\subsection{Contenedores Docker y reproducibilidad}

Docker permite ejecutar servicios en contenedores con configuración explícita. En el proyecto, Docker Compose levanta PostgreSQL 16 Alpine con variables de base de datos, puerto local, volumen persistente y \textit{healthcheck}. Esto aporta reproducibilidad porque el equipo de desarrollo puede iniciar la dependencia principal de datos sin instalar manualmente PostgreSQL en el sistema operativo.

Conceptualmente, Docker no reemplaza el diseño de arquitectura, sino que lo apoya a nivel de infraestructura. La aplicación conserva sus capas de cliente, servidor y persistencia; Docker facilita que la capa de persistencia se ejecute de forma controlada. Esta decisión es coherente con el objetivo de reducir fricción ambiental durante desarrollo, pruebas y futuras etapas de despliegue.

\section{Antecedentes históricos o contextuales}

Los sistemas de punto de venta evolucionaron desde cajas registradoras orientadas al registro de importes hacia plataformas integradas que conectan ventas, inventario, usuarios, reportes y cumplimiento normativo. En una farmacia, esta evolución es especialmente relevante porque el producto vendido puede estar sujeto a control por lote, fecha de vencimiento, proveedor, precio, receta, devolución y auditoría. Por tanto, el problema no consiste únicamente en ``vender productos'', sino en mantener consistencia operativa entre transacciones comerciales y existencias físicas.

En el contexto tecnológico actual, las aplicaciones web administrativas tienden a separar cliente y servidor mediante APIs. Esta separación permite que el frontend evolucione como interfaz especializada y que el backend mantenga reglas de negocio, seguridad y persistencia. El proyecto analizado adopta esta orientación: el cliente React consume endpoints bajo \texttt{/api}, el servidor Express organiza los módulos y PostgreSQL conserva la información transaccional.

La adopción de TypeScript tanto en cliente como en servidor responde a la necesidad de reducir errores en un sistema con varias fronteras de datos. De forma complementaria, el paquete compartido con esquemas Zod permite que la validación no dependa solamente de formularios visuales ni de supuestos informales. Esta decisión es relevante para la rigurosidad del sistema, porque las entradas de usuario y las respuestas de API se contrastan con contratos explícitos.

El antecedente técnico inmediato del proyecto es una primera base funcional con autenticación, roles, permisos y auditoría. Aunque varios módulos del dominio farmacéutico aún aparecen como estructura planificada, el diseño inicial ya establece los límites arquitectónicos sobre los cuales se podrán implementar compras, inventario, ventas, caja, facturación, devoluciones y reportes. Esta base resulta coherente con una estrategia incremental: primero se estabilizan fundamentos transversales, luego se incorporan procesos de negocio de mayor complejidad.

# Vite React UI Architecture Patterns

Este documento define lineamientos de arquitectura para una aplicación React nueva creada con Vite. Es agnóstico al dominio: los nombres de carpetas, módulos y ejemplos deben adaptarse al producto, pero las responsabilidades y límites entre capas deben mantenerse.

## Principios

- El código debe estar organizado por responsabilidad, no por conveniencia de import.
- Las páginas componen flujos; los módulos encapsulan datos y comportamiento portable; los componentes renderizan UI.
- Las APIs de bajo nivel no deben filtrarse hacia páginas o componentes de presentación.
- Los módulos deben ser profundos internamente, pequeños hacia afuera y reutilizables fuera de la app web.
- El código compartido debe ser realmente reutilizable y no contener lógica específica de pantallas.
- Los valores semánticos deben tener nombre: evita magic numbers, magic strings y colores crudos.
- El estado global debe leerse con selectores estables, no con objetos completos.

## Estructura Recomendada

Forma recomendada para un proyecto Vite:

```txt
src/
  api/
  clients/
  components/
  constants/
  hooks/
  i18n/
  layouts/
  modules/
  pages/
  routes/
  shared/
  test/
  ui/
  utils/
```

Responsabilidades:

- `src/pages`: pantallas asociadas a rutas.
- `src/routes`: definición de rutas, guards y route-level composition.
- `src/layouts`: shells persistentes, navegación y regiones comunes.
- `src/modules`: capa portable de datos por feature: contratos, APIs, facades, hooks de datos, stores, schemas y utils. Debe poder empaquetarse para otra UI, incluso React Native.
- `src/components`: componentes reutilizables dentro de la app.
- `src/ui`: primitivas visuales, tokens, utilidades de UI y wrappers genéricos.
- `src/shared`: DTOs, entidades, enums, constantes y utilidades puras compartidas por más de una feature.
- `src/api`: cliente base, manejo común de errores, auth headers y configuración HTTP.
- `src/clients`: clientes técnicos no ligados a una feature, como sockets, storage o bridges externos.

Reglas:

- No pongas lógica de pantalla dentro de `src/ui`.
- No pongas componentes React dentro de `src/shared`.
- No pongas componentes React, JSX, estilos, rutas, layouts, iconos, copy de pantalla ni dependencias DOM dentro de `src/modules`.
- No dupliques primitivas genéricas en `src/components` si pertenecen al sistema de UI.
- No muevas código a `src/shared` hasta que exista reutilización real.

## Módulos De Feature

Cada feature debe vivir en `src/modules/<feature>` y exponer una interfaz pública pequeña desde `index.ts`. Un módulo es una capa de datos portable: debe ser consumible por una app web, una app React Native u otra interfaz sin arrastrar UI web.

Forma recomendada:

```txt
src/modules/<feature>/
  api/
  constants/
  facades/
  hooks/
  schemas/
  store/
  types/
  utils/
  index.ts
```

Responsabilidades:

- `api/`: clientes HTTP, WebSocket, SignalR, adapters o integraciones externas. Solo transporte.
- `constants/`: límites, claves de dominio, nombres de eventos y valores semánticos de datos. No copy visible ni labels de UI.
- `facades/`: operaciones de aplicación que coordinan API, store, normalización o persistencia.
- `hooks/`: hooks de datos consumidos por páginas, layouts o componentes app-level. Pueden usar React/Zustand, pero no DOM, router ni componentes.
- `schemas/`: validación, coerción y normalización con Zod u otra librería.
- `store/`: estado, acciones, selectores e implementación del store.
- `types/`: tipos propios de la feature que no pertenecen a `src/shared`.
- `utils/`: helpers puros, mappers y construcción de payloads.
- `index.ts`: barrel público de la feature.

Reglas:

- Importa desde el barrel del módulo siempre que el consumidor esté fuera del módulo.
- Evita imports profundos hacia `modules/<feature>/api`, `store` o `utils` desde páginas.
- Mantén `api/` libre de modelos de UI, localStorage, env readers, error mapping y normalización local.
- Si una pantalla necesita una operación compleja, crea un hook o facade de módulo.
- No crees carpetas `components/`, `pages/`, `layouts/`, `routes/`, `styles/` ni archivos `.tsx` dentro de `src/modules`.
- No importes desde `src/components`, `src/ui`, `src/pages`, `src/layouts`, `react-router`, librerías de iconos, CSS/Tailwind o APIs DOM dentro de `src/modules`.
- No pongas textos visibles de producto, labels, placeholders, tooltips ni mensajes de layout en `src/modules`; ese copy vive en `pages`, `components` o i18n.
- No uses storage de navegador directamente en `src/modules`. Si un store necesita persistencia, usa una clave/contrato portable y configura el adapter desde una capa app-level o `src/clients`.
- El barrel `index.ts` de un módulo solo debe exportar hooks de datos, facades, stores reset/selectors públicos, tipos, schemas y constantes de dominio. Nunca exporta componentes.

Flujo preferido:

```txt
page -> module hook/facade -> store/api
```

## Páginas, Layouts Y Componentes

### Páginas

Las páginas orquestan composición de ruta.

Pueden tener:

- estado local de presentación;
- lectura de URL params o search params;
- composición de secciones;
- conditional rendering específico de ruta.

No deben tener:

- clientes HTTP directos;
- reglas de negocio reutilizables;
- normalización de responses;
- coordinación de estado compartido entre pantallas.

### Layouts

Los layouts definen shell, navegación, providers visuales y regiones persistentes.

Reglas:

- El layout no debe conocer detalles internos de una feature salvo módulos app-level.
- El shell global debe renderizar providers y UI persistente como sidebar, header, footer o toaster.
- Usa rutas anidadas para separar shell de pantalla.

### Componentes

Usa estos límites:

- `src/components`: UI reutilizable dentro de la app, pero no genérica para el sistema de UI.
- `src/pages/<feature>` o `src/pages/<feature>-page.tsx`: UI acoplada a una pantalla o feature específica.
- `src/ui`: primitivas genéricas, tokenizadas, accesibles y reutilizables.

Reglas:

- Si renderiza JSX, usa clases CSS/Tailwind, importa iconos, lee router params o muestra copy, no pertenece a `src/modules`.
- Una pantalla puede importar hooks/facades desde `src/modules/<feature>` y convertir esos datos en UI local.
- Un componente reutilizable puede importar hooks de módulo solo cuando representa una pieza app-level, pero no debe importar internals profundos del módulo.

## API Y Clientes

Separar clientes base de clientes de feature.

```txt
src/api/
  axiosApi.ts
  ApiError.ts
  authTokenProvider.ts
src/modules/orders/api/
  ordersApi.ts
```

Reglas:

- Autenticación, locale, session headers y manejo base de errores van en el cliente global.
- Los clientes de feature solo construyen endpoints, pasan `params`/payloads y devuelven `response.data`.
- Request builders, mappers, error messages de producto y normalización van en `utils`, `schemas` o `facades`.
- No llames APIs de feature directamente desde páginas si la operación pertenece al comportamiento del módulo.
- Los clientes `api/` no parsean formularios, no conocen copy de usuario, no actualizan stores y no deciden navegación.
- Los errores técnicos se transforman en el cliente base; los mensajes de experiencia de usuario se deciden en hooks/facades app-level o UI.

Ejemplo:

```ts
// modules/orders/api/ordersApi.ts
export const ordersApi = {
  getAll: async (params?: OrderQueryDto): Promise<PaginatedResult<Order>> => {
    const response = await axiosApi.get<PaginatedResult<Order>>(ORDERS_BASE_PATH, { params })
    return response.data
  },
}
```

## Facades Y Hooks De Aplicación

Usa facades cuando una operación coordina varias piezas. Las facades son la frontera de casos de uso del módulo: pueden llamar APIs, stores, schemas, mappers y clientes técnicos, pero no renderizan UI ni importan componentes.

Buen uso de facade:

- cargar una entidad y registrar acceso reciente;
- combinar datos de dos endpoints;
- aplicar normalización antes de actualizar store;
- resolver una acción de detalle desde URL;
- centralizar una mutación con optimistic update o versionado.

Usa hooks cuando la feature necesita una interfaz React de datos. El hook puede seleccionar estado, exponer acciones estables, disparar cargas y adaptar errores a un contrato de datos; no debe leer router params, renderizar JSX ni contener copy visual.

```ts
export function useOrderDetailShell() {
  const detail = useOrderDetail()
  const [searchParams] = useSearchParams()
  const orderId = searchParams.get('order')

  // Route-level synchronization belongs here, not in the page.

  return detail
}
```

## Zustand Store Pattern

Los stores deben separar estado, acciones, selectores e implementación. El store es parte de `src/modules`, por lo tanto debe ser portable y no depender de UI web.

```txt
store/
  OrdersState.ts
  OrdersActions.ts
  OrdersSelectors.ts
  OrdersStore.ts
  index.ts
```

Reglas:

- Define `State` y `Actions` como interfaces separadas.
- Exporta `initialState` desde el archivo de estado.
- Usa selectores pequeños y estables.
- Lee desde el store mediante selectores, no mediante el objeto completo.
- Usa `useShallow(...)` cuando un selector devuelva objetos o arrays.
- No devuelvas fallback arrays u objetos frescos desde selectores; define constantes de módulo.
- Las acciones públicas del store orquestan flujos de datos y delegan transporte a facades o APIs; no deben navegar, mostrar toasts, abrir modales ni renderizar UI.
- Las acciones destructivas deben ser conservadoras; evita optimistic updates en deletes salvo que exista recuperación clara.
- Los stores no importan componentes, `react-router`, CSS, iconos ni helpers de UI.
- Si usas persistencia, aísla nombres de claves y adapters para no acoplar el store a `window.localStorage`.

Ejemplo:

```ts
const EMPTY_ROWS: OrderRow[] = []

export const selectOrderRows = (state: OrdersStore) => {
  return state.rowsByOrderId[state.activeOrderId ?? ''] ?? EMPTY_ROWS
}
```

Evita:

```ts
const rows = useOrdersStore((state) => state.rowsByOrderId[id] ?? [])
const store = useOrdersStore()
useEffect(() => {}, [store])
```

Prefiere:

```ts
const rows = useOrdersStore(selectOrderRows)
const doLoadOrder = useOrdersStore((state) => state.doLoadOrder)
useEffect(() => {
  void doLoadOrder(id)
}, [doLoadOrder, id])
```

## Constantes Y Valores Semánticos

Centraliza:

- claves de `localStorage`;
- límites de paginación;
- duraciones de debounce;
- timeouts y animation timings;
- nombres de eventos;
- fragmentos de rutas;
- nombres de query params;
- duración de toasts.

Dentro de `src/modules`, estas constantes deben ser de dominio/datos. Labels de tablas, copy de botones, placeholders, tooltips, textos de empty states y variantes visuales viven fuera de `modules`, en `pages`, `components` o i18n.

Ejemplo:

```ts
export const DEFAULT_TOAST_DURATION_MS = 4500
export const ORDERS_BASE_PATH = '/orders'
export const ORDER_ID_SEARCH_PARAM = 'order'
```

## Diseño, Tokens Y Colores

No hardcodees colores en páginas o componentes. Los módulos no deben contener UI ni estilos.

Evita:

```tsx
<div className="bg-[#101828] text-white border-slate-700" />
```

Prefiere:

```tsx
<div className="bg-background text-foreground border-border" />
```

Reglas:

- Los componentes consumen tokens semánticos.
- Los presets de tema pueden definir valores crudos; la UI no.
- Si falta un color, agrega un token o deriva uno en el sistema de tema.
- Usa primitivas accesibles del sistema de UI antes de crear variantes locales.
- Usa iconos de una librería consistente para botones de herramientas y acciones conocidas.

## Theming

Los tokens base viven en el sistema de UI. Los presets propios de la app viven fuera de `src/modules` si contienen UI, CSS variables o detalles de plataforma.

```txt
src/ui/globals.css
src/clients/theme/
  hooks/
  presets/
  store/
  utils/
  index.ts
```

Reglas:

- El tema debe actualizar CSS variables, no intercambiar hojas completas.
- Soporta `light`, `dark` y `system` si existe selección de tema.
- Persiste preferencias sin sobrescribir configuraciones previas del usuario.
- La inicialización de tema debe bloquear pantallas solo el mínimo necesario para evitar parpadeos.
- No crees módulos de tema dentro de `src/modules` cuando dependan de DOM, CSS variables, `matchMedia`, `document` o storage web.

## i18n

Reglas:

- El código, identificadores y keys deben estar en inglés.
- Los valores traducidos deben vivir por locale y namespace.
- Inicializa i18n una sola vez en `src/i18n/index.ts`.
- Persiste la selección de idioma en `localStorage` o preferencias de usuario.
- No hardcodees copy de producto dentro de componentes reutilizables.

Forma recomendada:

```txt
src/i18n/
  index.ts
  languageStorage.ts
  locales/
    en/
      common.json
      navigation.json
    es/
      common.json
      navigation.json
```

## Formularios Y Validación

Reglas:

- Usa schemas para validación y normalización.
- Mantén schemas de datos cerca del módulo que los consume.
- Mantén schemas puramente visuales o de formularios complejos junto a la página si solo sirven a una pantalla.
- Evita `watch()` de React Hook Form cuando `useWatch({ control, name })` resuelve el caso.
- Mantén field components reutilizables libres de copy específica de una feature.
- Las páginas no deben construir payloads complejos inline; usa mappers o facades.

## UI App-Level

Comportamientos globales de UI deben vivir fuera de `src/modules`, por ejemplo en `src/components`, `src/ui`, `src/layouts`, `src/clients` o una carpeta app-level dedicada.

Ejemplos:

- toasts;
- drawers globales;
- modales de sesión;
- estado de tabs visuales;
- timers de animación app-level.

Reglas:

- Renderiza el proveedor visual global una vez en el app shell.
- Dispara comportamientos mediante hooks del módulo UI.
- Centraliza tipos, posiciones, duraciones y defaults.

## Rutas Y Deep Links

Reglas:

- Las rutas declaran composición; la lógica de hidratación de URL vive en páginas o hooks app-level fuera de `src/modules`.
- Si una pantalla usa query params para abrir una entidad o tab, el refresh del navegador debe restaurar el estado.
- Los route params y search params deben tener constantes semánticas.
- Los errores esperados de acceso o entidad inexistente deben resolverse con navegación clara.

## Logging Y Errores

Reglas:

- No uses `console.*` directamente en la app.
- Usa una interfaz de logging propia: `Logger.error`, `Logger.warn`, `Logger.info`, `Logger.debug`.
- Centraliza traducción de errores técnicos a mensajes de usuario.
- Los clientes base pueden capturar errores técnicos; las facades o hooks deciden experiencia de usuario.
- Las facades y stores pueden exponer códigos/estados de error; la traducción final a copy visible pertenece a UI o i18n.

## Dependencias De UI

Reglas:

- Al agregar una primitiva basada en Radix, Base UI, shadcn u otra librería, declara explícitamente su dependencia.
- No asumas que un generador agregó peer dependencies o package dependencies correctamente.
- Mantén iconos, tablas, virtualización, forms y charts como decisiones explícitas de arquitectura.

## Testing Y Validación

Prioriza pruebas donde haya riesgo real:

- stores y selectors para estado compartido;
- hooks que sincronizan URL, persistencia o sockets;
- facades que orquestan mutaciones;
- componentes con estados vacíos, loading, error y permisos;
- rutas críticas y deep links.

Validación recomendada:

- lint para estilo y hooks;
- typecheck del proyecto o del scope tocado;
- tests unitarios o de integración por módulo;
- QA manual solo para flujos visuales o navegación crítica.

## Antipatrones

Evita:

- páginas que importan `featureApi` directamente;
- componentes, JSX, CSS, Tailwind, iconos o router dentro de `src/modules`;
- labels de pantalla, placeholders o copy de producto dentro de `src/modules`;
- stores que navegan, muestran toasts o abren modales;
- facades que retornan modelos de UI o dependen de componentes;
- stores leídos completos en muchos componentes;
- selectores que crean arrays u objetos nuevos como fallback;
- componentes que mezclan fetch, layout, validación y copy;
- colores crudos en clases Tailwind;
- strings de rutas, eventos o localStorage inline;
- lógica de negocio dentro de componentes del sistema de UI;
- `src/shared` convertido en un cajón de sastre;
- barrel files que exportan detalles internos innecesarios;
- efectos con dependencias de objetos completos cuando bastan acciones estables y valores escalares.

## Checklist Para Una Feature Nueva

- Crear `src/modules/<feature>/index.ts`.
- Definir constantes semánticas de rutas, keys, límites y eventos.
- Crear cliente en `api/` si hay transporte externo.
- Crear schemas para formularios, payloads o normalización.
- Crear facade si la operación coordina más que un request simple.
- Crear store separado en state, actions, selectors e implementación si hay estado compartido.
- Crear hooks de módulo para consumo desde páginas.
- Crear componentes o pantallas fuera de `src/modules`, en `src/pages` o `src/components`.
- Exportar solo la interfaz pública desde el barrel.
- Verificar que `src/modules/<feature>` no contenga `.tsx`, `components/`, imports de UI, imports de router ni copy visual.
- Escribir pruebas para selectors, hooks o facades de mayor riesgo.

## Regla De Decisión Rápida

- Si es una pantalla: `pages`.
- Si es comportamiento de dominio, datos o feature portable: `modules/<feature>`.
- Si es UI reutilizable dentro de una app: `src/components`.
- Si es UI genérica y tokenizada: `src/ui`.
- Si es contrato o utilidad pura compartida entre features: `src/shared`.
- Si es transporte externo: `modules/<feature>/api` o `src/api` para cliente base.
- Si es coordinación de datos: `facades` o hooks de módulo.
- Si es coordinación de UI, rutas, layout, URL o copy visible: `pages`, `components` o hooks app-level fuera de `modules`.
